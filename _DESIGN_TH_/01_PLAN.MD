# แผนการดำเนินงาน (Implementation Plan)

## ภาพรวมโครงการ (Project Overview)
แอปพลิเคชัน GUI ประสิทธิภาพสูงสำหรับการค้นหาไฟล์และโฟลเดอร์โดยใช้เอนจินความคล้ายคลึงกันที่ใช้เวกเตอร์ที่กำหนดเอง แอปพลิเคชันแสดงความสัมพันธ์ของไฟล์และความคล้ายคลึงกันของเวกเตอร์ด้วยกราฟโหนดสไตล์ Obsidian

## ฟีเจอร์หลัก (Core Features)
1. **ตัวรวบรวมข้อมูลระบบไฟล์ (File System Crawler)**: สแกนไดเรกทอรีและไดรฟ์ที่กำหนดโดยผู้ใช้แบบเรียกซ้ำ (เช่น `C:\`, `D:\`) โดยเคารพไฟล์ `.ignore` การตั้งค่าการข้ามเริ่มต้นจะข้ามโฟลเดอร์ `.git` และเส้นทางระบบ Windows (เช่น `C:\Windows`, `C:\System Volume Information`)
2. **ตัวสกัดเวกเตอร์ (Vector Extractor)**: คำนวณเวกเตอร์ตัวเลข (สูงสุด 42 มิติ) ตามคุณสมบัติของไฟล์ (เส้นทาง, ขนาด, การเข้ารหัส, ประเภท ฯลฯ)
3. **เอนจินค้นหาความคล้ายคลึงกัน (Similarity Search Engine)**: ค้นหาไฟล์โดยเปรียบเทียบเวกเตอร์หลายมิติ
4. **GUI กราฟแบบโต้ตอบ (Interactive Graph GUI)**:
   - มุมมองที่ 1: กราฟอิงตามเส้นทางจริง (Physical path) แสดงการเชื่อมโยงของเส้นทาง และเชื่อมโยงไปยังไฟล์/โฟลเดอร์ที่คล้ายกันมากในระยะทางใกล้ที่ระบุ
   - มุมมองที่ 2: กราฟความคล้ายคลึงกันของเวกเตอร์ แสดงการเชื่อมโยงของความคล้ายคลึงกันของเวกเตอร์ใกล้เคียง (เพื่อนบ้านที่ใกล้ที่สุด)
5. **การป้องกันโหนดทับซ้อน & การตั้งค่าความหนาแน่น**: รวมแถบเลื่อนสำหรับ "แยกมากขึ้น" (การผลักระยะห่างทั่วโลก) และ "แยกใกล้" (ตัวคูณการแพดเพื่อหลีกเลี่ยงการชนกัน/การทับซ้อนกันเฉพาะที่) เพื่อขยายระยะห่างของโหนดแบบไดนามิกและป้องกันชื่อทับซ้อนกัน
6. **ตัวควบคุมการสั่นกราฟ (Shake Graph Jitter Controls)**: ตัวเลือกแบบโต้ตอบในการเขย่าทุกโหนด (สั่นสะเทือนทั่วโลก), เขย่ากลุ่มที่เลือก (สั่นสะเทือนบริเวณใกล้เคียง), หรือเขย่าโหนดที่ทับซ้อนกัน (สั่นสะเทือนตามความใกล้ชิด) เพื่อสลายกลุ่มที่หนาแน่น
7. **รองรับหลายไดรฟ์และไดเรกทอรี**: เพิ่มเส้นทางราก (root paths) หรือตัวอักษรไดรฟ์ทั้งหมดลงในดัชนีการค้นหา
8. **การรายงานความคืบหน้าและ UI แบบไม่บล็อก**: เมื่อป้อนเส้นทาง ระบบจะสแกนและคำนวณเวกเตอร์บนเธรดพื้นหลัง ความคืบหน้าแบบเรียลไทม์จะสะท้อนใน GUI
9. **บอร์ดกระบวนการ Kanban ขณะทำงาน (Run-time Kanban Process Board)**: แทนที่จะเป็นแถบความคืบหน้าธรรมดา UI การนำเข้า/ทำดัชนีจะแสดงบอร์ด Kanban แบบเรียลไทม์ที่แสดงไฟล์ที่เคลื่อนผ่านคอลัมน์การประมวลผล (ค้นพบ, สกัด, กำลังเขียน, ทำดัชนีแล้ว)
10. **บันทึกการตรวจจับการแครชและค้าง (Crash & Hang Detection Log)**: ไฟล์บันทึกเฉพาะ (`import_progress.log`) จะติดตามขั้นตอนการทำดัชนีเพื่อวิเคราะห์เหตุการณ์การแครชหรือค้าง
    - ล้างข้อมูลอัตโนมัติ (ตัดทิ้งให้ว่างเปล่า) เมื่อแอปพลิเคชันเริ่มทำงาน
    - เขียนวันที่และเวลาเริ่มต้นเป็นบรรทัดแรกของบันทึกในการเริ่มการนำเข้าใหม่ทุกครั้ง
11. **แดชบอร์ดทรัพยากรเรียลไทม์ (Real-time Resource Dashboard)**: หน้าจอ GUI แสดงสถิติประสิทธิภาพของระบบระหว่างการทำงาน:
    - **จำนวนโหนดทั้งหมด (Total Nodes)**: ไฟล์/โฟลเดอร์ทั้งหมดที่โหลด/จัดทำดัชนีในปัจจุบัน แบ่งเป็นการคำนวณกับการเรนเดอร์ (เช่น `115,240 ทั้งหมด (เรนเดอร์ 1,200)`)
    - **การใช้หน่วยความจำ (Memory Usage)**: การใช้ RAM ปัจจุบันของกระบวนการแอปพลิเคชัน
    - **การใช้ดิสก์ (Disk Usage)**: การใช้งานพื้นที่เก็บข้อมูลทั้งหมดของไดรฟ์โฮสต์ที่ถูกสแกน
12. **การปรับความหน่วง UI แบบแปรผัน (Adaptive UI Throttling)**: ตรวจสอบความหน่วงของลูปกิจกรรม UI และโหลด CPU/IO โดยรวม หากการตอบสนองลดลง จะชะลอเธรดคนงานที่รวบรวม/จัดทำดัชนีแบบไดนามิก
13. **การเพิ่มประสิทธิภาพกราฟสเกลสูง (High-Scale Graph Optimization - 100k+ โหนด)**: การจัดทำหน้าต่าง LIMIT ระดับ SQL แบบไดนามิกและการจัดอันดับความเกี่ยวข้อง (คะแนนความสำคัญ) จำกัดการเรนเดอร์ GUI ให้เหลือเพียง $N_{\text{max}}$ โหนด รวมถึงการคัดแยก frustum ของมุมมองส่วนหน้าจอ การฉลากตามระดับรายละเอียด (LoD) การซูม และการแช่แข็งฟิสิกส์อัตโนมัติเมื่อจำนวนโหนดเกิน 1,500

## แผนการดำเนินงานทีละขั้นตอน (Step-by-Step Plan)
1. **ระยะที่ 1: รากฐานและตัวรวบรวมข้อมูล**
   - สร้างตัวเดินระบบไฟล์แบบหลายเธรดพร้อมการจัดการคอขวด CPU
   - เพิ่มลอจิกวิเคราะห์ไฟล์ `.ignore` (เช่นเดียวกับรูปแบบการสแกนแบบ `.gitignore`) และตั้งค่าข้ามเริ่มต้นไว้แล้ว
   - ตั้งค่าระบบสังเกตการณ์การเปลี่ยนแปลงไดเรกทอรีส่วนเพิ่ม (watcher) โดยใช้ Win32 API (`ReadDirectoryChangesW`) หรือ `watchdog`
   - ทำให้กระบวนการเธรดทำงานพื้นหลังเสร็จสิ้นโดยใช้ `QThread` / `QRunnable` ของ Qt ดังนั้นการสร้างดัชนีจะไม่บล็อกเหตุการณ์ GUI
2. **ระยะที่ 2: ระบบบันทึกและตรวจจับการแครช**
   - นำไปใช้งานคลาส `ImportLogger` ซึ่งจะล้างข้อมูล `import_progress.log` ตอนเริ่มต้นและต่อท้ายสแตมป์เวลาในกรณีที่เปิดทำงานนำเข้าใหม่
   - บันทึกเป้าหมายสำคัญเพื่อตรวจจับความบกพร่องที่ทำให้ค้าง
3. **ระยะที่ 3: เอนจินเวกเตอร์**
   - ควบคุมการแสดงผลเชิงคณิตศาสตร์ของคุณสมบัติทั้ง 42 ประการ
   - การดึงลักษณะจากที่อยู่ไฟล์, ตำแหน่งตัวแม่และเมทาดาตา
   - ปรับค่าความเสถียรของคุณสมบัติ (เช่น log-scale สำหรับไฟล์, ทำให้สแตมป์เวลากลับมาปกติ, เข้ารหัสแบบ one-hot กับส่วนขยาย)
4. **ระยะที่ 4: ฐานข้อมูลและการจัดเก็บ**
   - จัดตั้งฐานข้อมูล SQLite ในเครื่องสำหรับจัดเก็บเมทาดาตาและการเขียนกราฟเวกเตอร์
   - สร้างดัชนีเฉพาะสำหรับการดึงตำแหน่งโฟลเดอร์แบบย่อยที่รวดเร็วขึ้น
5. **ระยะที่ 5: ตรรกะการค้นหาและดูแลทรัพยากร**
   - การคำนวณระยะห่างเพื่อหากลุ่มเป้าหมาย (Cosine Similarity และ Euclidean Distance)
   - ควบคุมดัชนีเชิงพื้นที่ (ตัวอย่างเช่น KD-Tree หรือ HNSW) เพื่อเร่งความเร็วในการสอบถาม
   - บูรณาการการตรวจสอบทรัพยากรระบบใช้ `psutil` หรือใช้ API จากเครื่องโดยตรง
6. **ระยะที่ 6: GUI, กราฟ, & มุมมองความคืบหน้าแบบ Kanban**
   - พัฒนา PyQt6 / PySide6 ประกอบด้วยแถบค้นหา, ผู้จัดการโฟลเดอร์, และมุมมองกราฟคู่ซึ่งสลับได้
   - พัฒนา **Run-time Kanban Process Board widget** ให้ซ้อนกันและจัดการตัวนับไฟล์ผ่านคอลัมน์ได้ง่าย
   - เพิ่มสถานะบอกระดับด้านล่าง เพื่อแสดง: **จำนวนโหนด (เปรียบเทียบการคำนวณกับการเรนเดอร์)**, **การใช้งานแรม (RAM)**, และ **ความจุที่เหลือในดิสก์**
   - ควบคุมแถบสไลด์ใน GUI สำหรับ **แยกมากขึ้น (Global Spacing)** และ **แยกใกล้ (Collision/Overlap)** ที่ปรับค่าข้อมูลโดยอัตโนมัติจาก Javascript ผ่านหน้า WebEngine
   - เพิ่มแถบเพื่อปรับ **ขีดจำกัดโหนดที่เรนเดอร์ได้สูงสุด ($N_{\text{max}}$)** เพื่อการตัดข้อมูลทิ้งของส่วนแบ็คเอนด์
   - สร้างปุ่มควบคุมเพื่อ **[สั่นทั้งหมด]**, **[สั่นส่วนที่เลือก]**, และ **[สั่นสิ่งที่ทับซ้อน]** เป็นอินเทอร์เฟซผู้ใช้
   - เพิ่มแถบควบคุมระยะห่างเพื่อกำหนด “ใกล้” กับลิงก์ความคล้ายกันของไฟล์ 
   - นำเข้าฉลากแบบซูมตามระดับรายละเอียด (LoD) และปิดระบบฟิสิกส์อัตโนมัติใน Javascript
   - ติดตั้งตัวเลื่อนแบบเมาส์และการเลื่อนหน้าแบบแป้นพิมพ์ WASD/ลูกศร พร้อมกับปรับค่าความเร็วการเลื่อนที่ควบคุมผ่านความละเอียดซูม
7. **ระยะที่ 7: ปรับแต่งประสิทธิภาพแบบปรับตัว (Adaptive Performance Tuning)**
   - นำมอนิเตอร์สถานะและการเต้นหัวใจกลับเข้าไปยังกิจกรรมในลูปเหตุการณ์ของ GUI 
   - หากเธรดหลักที่ตรวจสอบมีความหน่วงมากกว่า 50 มิลลิวินาที ระบบจะส่งสัญญาณเตือนไปที่เธรดคนงานเพิ่มระยะเวลาในการประวิง
   - ใช้หลักการย่อหรือบดบังโหนดเฉพาะหน้า เมื่อข้อมูลมากเกินไปเพื่อป้องกันการแครชของหน้าจอผู้ใช้
