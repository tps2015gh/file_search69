# นโยบายการพัฒนา (Development Policy)

เอกสารนี้ให้รายละเอียดเกี่ยวกับข้อกำหนดของสภาพแวดล้อม การเลือกภาษา/เทคโนโลยี และกฎการเขียนโค้ดสำหรับโปรเจกต์

---

## 1. สภาพแวดล้อม & แพลตฟอร์มเป้าหมาย (Environment & Target Platform)
- **ระบบปฏิบัติการเป้าหมายหลัก (Primary Target OS)**: Windows 11.
  - การพัฒนาจะต้องให้ความสำคัญกับโครงสร้างไดเรกทอรีของ Windows 11, เส้นทาง (เช่น แบ็กสแลช vs ฟอร์เวิร์ดสแลช), และคุณลักษณะของไฟล์ระบบ (ซ่อน, ระบบ, ชั่วคราว)
  - แอปพลิเคชันต้องรองรับการสแกนอักษรไดรฟ์ราก (เช่น `C:\`, `D:\`, เป็นต้น) และจัดการข้อผิดพลาดเฉพาะที่เกิดจากไดรฟ์
- **ระบบปฏิบัติการเป้าหมายรอง (Secondary Target OS)**: macOS, Linux (ต้องรักษาความเข้ากันได้ข้ามแพลตฟอร์มเท่าที่เป็นไปได้ แต่ให้ความสำคัญกับ Windows 11 เป็นหลัก).

---

## 2. กลุ่มเทคโนโลยี & การเลือกภาษา (Technology Stack & Language Selection)
- **ภาษาโปรแกรม (Programming Language)**: **Python (เวอร์ชัน 3.10+)**
  - **เฟรมเวิร์ก GUI (GUI Framework)**: `PySide6` (Qt สำหรับ Python). จัดเตรียมวิดเจ็ต UI ที่ดูเหมือนแบบเนทีฟ สไตล์แบบ CSS (QSS) และเครื่องมือการจัดการเธรดที่แข็งแกร่ง
  - **การคำนวณเวกเตอร์ (Vector Computations)**: `NumPy` & `SciPy` (KD-Tree) สำหรับระยะทางของเมทริกซ์ความเร็วสูง
  - **เหตุผล (Reasoning)**: ในขณะที่ Go (Golang) ให้การท่องไปยังไฟล์แบบหลายเธรดที่รวดเร็วมาก แต่ Python ให้ระบบนิเวศแบบรวมศูนย์สำหรับทั้งคณิตศาสตร์เวกเตอร์ขั้นสูง (NumPy) และการแสดงผล UI/กราฟที่ซับซ้อน การรวม Go และ Python นำมาซึ่งความซับซ้อนระหว่างกระบวนการ/C-binding การใช้ Python ร่วมกับการเรียกใช้ระบบที่ปรับให้เหมาะสมแล้ว (`os.scandir`) และคนงานเบื้องหลัง ทำให้เกิดสมดุลที่ดีที่สุดระหว่างความเร็ว ความสามารถทางคณิตศาสตร์ และความสมบูรณ์ของ GUI.

## 3. นโยบายละเว้นการรวบรวมข้อมูล (Crawler Ignore Policies)
เพื่อป้องกันการสแกนไฟล์ที่ทำให้ประสิทธิภาพลดลงหรือการละเมิดสิทธิ์การเข้าถึง ตัวรวบรวมข้อมูลต้องบังคับใช้กฎการละเว้น:

### A. เส้นทางการละเว้นเริ่มต้น (Default Ignore Paths)
ระบบจะต้องข้ามเส้นทางต่อไปนี้โดยค่าเริ่มต้น:
- โฟลเดอร์ `.git`
- เส้นทางระบบ Windows:
  - `C:\Windows`
  - `C:\System Volume Information`
  - `C:\$Recycle.Bin`
  - `C:\$WINDOWS.~BT`
  - `C:\$WinREAgent`
  - ไฟล์ระบบ Paging: `pagefile.sys`, `hiberfil.sys`, `swapfile.sys`
- โฟลเดอร์ชั่วคราว (ตัวอย่างเช่น โฟลเดอร์ Temp ใน AppData ของผู้ใช้)

### B. กฎการละเว้นที่กำหนดโดยผู้ใช้ (User-defined Ignore Rules)
- แอปพลิเคชันจะรองรับไฟล์ `.ignore` (วางไว้ในระดับรากของโฟลเดอร์ใดๆ ที่สแกน) ที่ซึ่งผู้ใช้สามารถระบุรูปแบบ glob (เช่น `*.tmp`, `node_modules/`, `dist/`).

## 4. นโยบายความปลอดภัย & ประสิทธิภาพ (Safety & Performance Policies)
เพื่อป้องกันไม่ให้ PC/Notebook ทำงานช้าลงและลูปการหยุดทำงาน (crash loops):

### นโยบาย A: การชะลอ CPU (CPU Throttling)
- เธรดตัวรวบรวมข้อมูลต้องหยุด (sleep) เป็นเวลาอย่างน้อย `5ms` หลังจากประมวลผลไฟล์ทุกกลุ่มละ `100` ไฟล์ เพื่อคืนเวลาการใช้งาน CPU กลับสู่ OS
- ห้ามเรียกใช้การเดินทางในระบบไฟล์ (filesystem walks) บนเธรดหลักของ UI โดยเด็ดขาด

### นโยบาย B: ความต้านทานข้อยกเว้น (Exception Immunity)
- การเดินทางในระบบไฟล์ต้องดักจับ `PermissionError`, `FileNotFoundError`, และ `OSError` อย่างนุ่มนวลโดยไม่หยุดกระบวนการทำดัชนี เส้นทางใดที่ล้มเหลวควรถูกบันทึกไว้อย่างเงียบๆ

### นโยบาย C: การเขียน SQLite แบบปลอดภัยต่อเธรด (Thread-Safe SQLite Writes)
- การทำงานของ SQLite ต้องใช้การเชื่อมต่อฐานข้อมูลแยกกันในแต่ละเธรด
- ตัวรวบรวมข้อมูลต้องเขียนข้อมูลเป็นธุรกรรม (ขนาดชุด: 1000 ไฟล์) แทนที่จะใช้คำสั่งเดี่ยวๆ เพื่อป้องกันปัญหาคอขวดในการเขียนดิสก์

### นโยบาย D: การชะลอแบบปรับตัว & ความเร็วในการตอบสนอง UI (Adaptive Throttling & UI Responsiveness)
- GUI ต้องมีการใช้งานตัวมอนิเตอร์ความเร็วในการตอบสนอง (responsiveness monitor) ที่ทดสอบความหน่วงลูปกิจกรรม (event loop latency)
- หากความหน่วงลูปกิจกรรมของเธรดหลักเกิน `50ms`, เธรดเบื้องหลังของคนเก็บข้อมูล (background crawler thread) จะถูกลดทอนประสิทธิภาพแบบไดนามิกโดยการเพิ่มระยะเวลาการชะลอในระดับไมโคร (micro-sleep duration).
- สัญญาณบอกความคืบหน้าแบบสด (Live progress indicators - เช่น แถบแสดงความคืบหน้าและแดชบอร์ดสถิติ) ต้องรับการปรับปรุงผ่านทางสัญญาณสレッド (thread signals) ที่ปลอดภัยเพื่อป้องกันสภาวะความขัดแย้ง (race conditions).

## 5. นโยบายการตรวจสอบ & ยืนยัน (Review & Verification Policy)
- **ข้อกำหนดการทดสอบหน่วย (Unit Testing Requirement)**: ทุกมิติการคำนวณทางคณิตศาสตร์ (0 ถึง 41) จะต้องมีการทดสอบระดับยูนิตตามลำดับเพื่อยืนยันขอบเขตของค่า `[0.0, 1.0]` หรือ `[-1.0, 1.0]` (อ้างอิงถึงข้อกำหนดของแต่ละมิติใน 04_SCHEMA.MD)
- **การตรวจสอบ (Validation)**: การจำลองการสแกนไดเรกทอรีในระบบไฟล์จำลอง (mock filesystem directory scans) จะต้องถูกนำมาใช้ทดสอบเพื่อป้องกันไม่ให้แอปพลิเคชันไปยึดติดอยู่กับไดเรกทอรีที่แท้จริงของผู้ใช้งาน.
