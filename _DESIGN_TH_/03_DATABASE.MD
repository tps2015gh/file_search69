# กลยุทธ์ฐานข้อมูล (Database Strategy)

## ภาพรวม (Overview)
เพื่อให้มั่นใจว่าเวลาในการค้นหาเร็วที่สุดเท่าที่เป็นไปได้และป้องกันคอมพิวเตอร์ทำงานช้าลง แอปพลิเคชันใช้กลยุทธ์การจัดทำดัชนีแบบผสมผสานโดยใช้ฐานข้อมูล SQLite ในระบบเพื่อเก็บข้อมูลเมทาดาตาเชิงสัมพันธ์ และดัชนีเวกเตอร์ในหน่วยความจำ (memory-mapped/in-memory) สำหรับการสืบค้นความคล้ายคลึงกัน

## กลุ่มเทคโนโลยี (Technology Stack)
1. **ฐานข้อมูล SQLite (SQLite Database)**
   - จัดเก็บเส้นทางไฟล์, คุณลักษณะของไฟล์, การตั้งค่าละเว้น (.ignore), และเวกเตอร์ที่คำนวณไว้ล่วงหน้า
   - ดำเนินการในโหมด Write-Ahead Logging (`WAL`) พร้อมตั้งค่า `synchronous = NORMAL` เพื่อให้แน่ใจว่าการเขียนข้อมูลเป็นไปอย่างรวดเร็วระหว่างการทำงานหนักของตัวรวบรวมข้อมูล โดยไม่ทำให้ GUI ค้าง
2. **ดัชนีความคล้ายคลึงกันในหน่วยความจำ (In-Memory Similarity Index)**
   - เก็บค่าการฝังเวกเตอร์ (อาร์เรย์ NumPy float32) โหลดไว้ในหน่วยความจำเป็นเมทริกซ์
   - เมื่อทำการค้นหา การคำนวณระยะทางจะถูกประมวลผลแบบคู่ขนานโดยใช้การดำเนินการแบบเวกเตอร์ของ NumPy (การคูณเมทริกซ์-เวกเตอร์สำหรับระยะโคไซน์)
   - สำหรับระดับสเกล (เช่น >100,000 ไฟล์) จะมีการสร้างโครงสร้าง SciPy KD-Tree เพื่อสืบค้นจุดใกล้เคียงภายในรัศมีแบบยุคลิดที่กำหนดไว้อย่างมีประสิทธิภาพ

## เทคนิคการเพิ่มประสิทธิภาพ (Optimization Techniques)
- **การแทรกข้อมูลจำนวนมาก (Bulk Insertions)**: ผลลัพธ์ของตัวรวบรวมข้อมูลจะถูกแทรกเป็นชุด (ตัวอย่างเช่น 500-1000 ระเบียนต่อหนึ่งธุรกรรม) เพื่อลดภาระ (overhead) ในการเขียนข้อมูล
- **การตั้งค่าสถานะการเปลี่ยนแปลง (Dirty Flagging)**: ไฟล์ที่จัดเก็บจะมีสถานะธง (flag) ว่า `dirty`. เมื่อตัวเฝ้าระวังระบบไฟล์รับรู้ถึงการแก้ไขของไฟล์ จะมีการอัปเดตเฉพาะคุณลักษณะ/เวกเตอร์ของไฟล์นั้นๆ
- **กราฟที่ถูกตัดแต่ง (Pruned Graphs)**: สำหรับการแสดงภาพผลลัพธ์ คำขอกราฟจะถูกตีกรอบไว้แค่ลำดับต้นๆ (top-K) ของไฟล์ที่คล้ายคลึงกัน (เช่น K=50) เพื่อป้องกันการโหลดภาพมากเกินไปบนแคนวาสของหน้าต่างแสดงผล

## การตัดสินใจเชิงสถาปัตยกรรม (Architectural Decision): ทำไมจึงไม่ใช้ ChromaDB?
หลักยึดเหนี่ยวที่สำคัญของโปรเจกต์นี้คือการปฏิเสธการใช้งานฐานข้อมูลเวกเตอร์โดยเฉพาะอย่าง ChromaDB และเลือกใช้ SQLite + NumPy/SciPy แทน
1. **การถ่วงน้ำหนักมิติแบบไดนามิก (Dynamic Dimension Weighting)**: ChromaDB (และดัชนีแบบ HNSW โดยทั่วไป) จำเป็นต้องล็อกระยะห่างตายตัวในขณะที่สร้างดัชนี แต่โปรเจกต์ของเราต้องมี "เมทริกซ์การถ่วงน้ำหนักความคล้ายคลึง" เพื่อให้ผู้ใช้สามารถปรับน้ำหนักได้แบบทันทีผ่านแถบเลื่อนบน GUI (เช่น ให้ความสำคัญกับวันเวลามากกว่านามสกุลไฟล์) กราฟ HNSW ไม่สามารถรองรับการปรับน้ำหนักมิติใหม่ได้หากไม่มีการสร้างดัชนีใหม่ทั้งหมด แต่ NumPy สามารถคำนวณในหน่วยความจำได้ทันที
2. **ภาระการเขียน (Write Overhead)**: SQLite ในโหมด WAL มีความเร็วสูงกว่ามากสำหรับการแทรกข้อมูลขนาดเล็กแบบเป็นชุดระหว่างการทำงานของตัวรวบรวมข้อมูล ในขณะที่ ChromaDB มีภาระเพิ่มเติมจากการบำรุงรักษากราฟ HNSW ซึ่งจะทำให้ตัวรวบรวมข้อมูลทำงานช้าลง
